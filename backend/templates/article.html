Cloudflare published an Agent Readiness Score. I've been trying to answer a different question.

Their tool asks if your site is configured to serve agents. Mine asks if you can identify agent traffic before it says a word.

Connor Jones 8 min read April 2026

Cloudflare published a post last week introducing a tool that checks whether your site is configured to work well with AI agents — an "agent readiness score." Does it have a robots.txt? Does it serve clean markdown when an agent asks for it? Does it expose an MCP server? All good signals, but every one depends on the agent or the server choosing to declare intent. The authors mention that as of February 2026, only 3 of 7 agents they tested actually request markdown by default, which left a gap big enough that they had to build a workaround for it.

I've been working on a different question. Instead of "is your site configured to welcome agents," I want to know: can you tell what kind of client is connecting to your site before it says anything at all? Before it sends an HTTP header. Before it identifies itself. Before it has the opportunity to tell you the truth or lie.

I expected to find that each AI SDK left its own mark — that traffic from an OpenAI client would look distinct from Anthropic, from LangChain, from anything else. It doesn't work that way. When I went looking, every Python AI SDK looked identical. The mark isn't left by the SDK. It's left by something lower down.

To explain why, I need to back up — because the thing doing the marking happens before most people think the conversation has even started.

The handshake happens before hello

When you connect to a website over HTTPS, there's a negotiation that happens before any content is exchanged. Both sides need to agree on how they're going to encrypt the connection. The client goes first.

That first message is called the ClientHello. It contains a list of encryption options the client supports, which extensions it understands, and what protocols it speaks. The server reads it, picks from the options, and the handshake continues from there. Only after all of that does your browser send an HTTP request.

Client Server │ │ │──── ClientHello ─────────────▶│ ← fingerprint lives here │◀─── ServerHello ──────────────│ │◀─── Certificate ──────────────│ │ ... handshake continues... │ │──── Finished ────────────────▶│ │◀─── Finished ─────────────────│ │ │ │──── GET / HTTP/1.1 ──────────▶│ ← application layer starts here

The contents of that ClientHello are controlled by the TLS library the client was compiled against — not the application running on top of it. A Python script using the Anthropic SDK and a Python script using the OpenAI SDK both go through the same underlying TLS library. They look identical at this layer. A Go program uses Go's own TLS implementation, which looks different from anything Python ships. curl on macOS links against LibreSSL; curl on Linux links against OpenSSL. Same tool, different library, different fingerprint.

JA4 is a hashing scheme that turns the contents of a ClientHello into a short, comparable string. That's the fingerprint. One more thing worth knowing: Chrome deliberately inserts random reserved values into its ClientHello — a mechanism called GREASE — to prevent servers from rejecting connections that contain unknown extensions. The presence of those values is itself a reliable signal. Nothing except a Chromium-family browser sends them.

"User-Agent strings are voluntary. GREASE values are not."

What the catalogue actually shows

All Python AI SDKs — OpenAI, Anthropic, LangChain — produce the same fingerprint on the same machine. Not similar. Identical. The SDK doesn't control the TLS handshake; the language does, specifically the TLS library it's built on. So what looks like "Anthropic SDK traffic" from a fingerprinting perspective is just Python traffic. The vendor is invisible at this layer.

What you can see is what's underneath the tool. The table below shows clients captured across Linux and macOS. The Signal column is the one that matters.

Two findings worth calling out. Go's TLS implementation is entirely its own — not OpenSSL, not BoringSSL — and nothing else in the catalogue resembles it. Unambiguous from the first packet. The other is curl: same tool, different operating system, completely different profile. A detection rule written for curl as a single fingerprint is missing half the traffic. The overlap between Rust and curl on Linux is real and worth naming — both link against OpenSSL, so they share cipher suite characteristics. In practice, ALPN presence is usually enough to separate them, but it's a reminder that this is probabilistic, not deterministic.

That same honesty applies to the agent/tool split you'll see on the /stats page. The classifier labels some traffic as "agent" (a Python HTTP client) and some as "tool" (curl). The actual basis for that distinction is one number: how many TLS extensions the client sends. curl typically sends 7 or 8. Python's HTTP library typically sends 9 to 12. That's the entire signal. There's no deeper fingerprint here — if someone ran curl with a few extra flags, or a Python script configured to be unusually minimal, the classifier would get it wrong. It's a reasonable default assumption, not a reliable identification.

The weakest boundary in the classifier is agent vs tool. Both curl and Python's requests use OpenSSL with similar cipher lists. The only signal is extension count: curl sends ≤8, Python's ssl module sends 9–12. One threshold. A curl invocation with extra flags or a minimal Python client configured without extras would cross it in the wrong direction. For fingerprints not in the catalogue, this is a heuristic guess — not a classification.

For fingerprints in the catalogue, the classification is an exact match. For everything else, it falls back to heuristics like these — and the gaps surface. The /stats page has a section that flags fingerprints seen repeatedly that the catalogue couldn't match exactly. Those are the ones worth capturing next.

This site serves agents differently

The terminal at the top of this page wasn't a mockup. When you loaded this article, a proxy sitting in front of the server classified your connection before the TLS handshake was complete — and routed you to a different backend based on what it found. If you're reading this in a browser, you got HTML. If you arrived via a Python HTTP client, a Go program, or curl, you got JSON.

Here's how that works.

visitor (TLS) │ ▼ ┌─────────────────┐ │ TLS proxy │ ← MSG_PEEK: reads ClientHello without consuming │ │ ← computes JA4, classifies client │ │ ← completes TLS handshake (same bytes) │ │ ← reads decrypted HTTP, injects X-Client-* headers └────────┬────────┘ │ plain HTTP (internal network) ▼ ┌─────────────────┐ │ routing decision│ └──┬──────────┬───┘ │ │ ▼ ▼ browser agent HTML JSON

Before the TLS handshake begins, a client sends a ClientHello — a plaintext packet that includes the cipher suites and extensions it supports. The proxy reads this packet using MSG_PEEK, a socket operation that copies bytes from the kernel buffer without consuming them. That's enough to compute the JA4 fingerprint and classify the client. The bytes then remain in the buffer, and the proxy proceeds with a normal TLS handshake — from the client's perspective, nothing unusual happened.

From there it's a standard terminating proxy. The proxy holds the TLS session, reads the decrypted HTTP request, injects classification headers, and forwards plain HTTP to the appropriate backend on the internal network. The classification — X-Client-Type, X-Client-JA4, X-Client-Confidence — arrives at the backend before a single line of application code runs.

The result is two separate backends: one that returns HTML for browsers, and one that returns structured JSON for everything else. Simulate what an agent receives by clicking below — the response is exactly what a Python HTTP client or curl would get when hitting this endpoint.

The agent_meta block at the bottom is worth reading. It tells the client exactly what was detected, at what layer, and why it received JSON instead of HTML. detection_layer: "network" isn't rhetorical — the routing decision was made from five fields in a TLS packet, before any HTTP was exchanged.

Two layers, two different problems

The two layers aren't competing approaches — they answer different questions. The application layer tells you whether your site is configured to serve agents. The TLS layer tells you what's actually connecting, regardless of what it claims. Neither is sufficient on its own. TLS identity without good application-layer serving is just a label. Application-layer readiness without identity is infrastructure that requires agents to opt in before it works.

There's a practical constraint worth naming if you're thinking about building something like this. Cloudflare Tunnel terminates TLS at Cloudflare's edge before the request reaches your origin — which means the ClientHello never arrives at your server. You can't fingerprint what you never see. Running classification at the TLS layer requires sitting before that termination point: either at the origin itself, or at a layer in your stack that receives the raw connection. That's an architectural decision you have to make up front.

What you get if you do is routing logic that doesn't require any application changes. The backend doesn't need to know how the classification works — it just reads a header. You can serve structured data to agents, rate-limit by client class, or log runtime version drift in your traffic patterns, all without touching application code. The proxy handles it before HTTP begins.
