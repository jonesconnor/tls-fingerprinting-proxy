Can You Identify an Agent Before It Speaks?

Their tool asks if your site is configured to serve agents. Mine asks if you can identify agent traffic before it says a word.

Connor Jones 8 min read April 2026

Cloudflare published a post last week introducing a tool that checks whether your site is configured to work well with AI agents. The tool is genuinely useful to an operator preparing a site for agent traffic: it audits the configuration choices on their side of the conversation, whether the site has a robots.txt, whether it serves markdown, and whether it exposes an MCP server. What it cannot audit is the client. The checklist only tells you whether you are ready to receive agents that cooperate. The authors mention that as of February 2026, only 3 of 7 agents they tested actually request markdown by default, which is the first sign that cooperation is not guaranteed.

The question I've been working on is whether you can tell what kind of client is connecting before it identifies itself, before it has sent a single HTTP byte.

Every HTTPS connection opens with a negotiation that happens entirely before application code runs on either side. That negotiation carries information the client cannot suppress. If those signals are consistent enough to classify, the identification doesn't depend on the client cooperating.

The Handshake

The client always starts the conversation. That opening message is called the ClientHello. It contains a list of encryption options the client supports, which extensions it understands, and what protocols it speaks. The server reads it, picks from the options, and the handshake continues from there. Only after all of that does an HTTP request get sent.

If those fields differ consistently from one client to the next, the ClientHello is enough to identify what's connecting before any application data is exchanged. To make that comparison consistent across clients, the field values get hashed into a short string — that's the fingerprint. The scheme this proxy uses is called JA4, an open specification that defines exactly which ClientHello fields to include and how to sort them before hashing.

This Site Serves Agents Differently

That fingerprint is what this site acts on. A proxy in front of this server reads it during the TLS handshake, classifies the connection as browser or non-browser, and routes accordingly. Browsers receive HTML; everything else receives structured JSON. The terminal at the top of the page surfaces the classification details for your specific connection.

Because JA4 is an open standard, a public database of known fingerprints exists — ja4db — mapping hashes to identified clients: browsers, command-line tools, HTTP libraries. On startup, the proxy downloads this database in full and indexes it in memory, so every connection gets a lookup against thousands of known fingerprints without adding any per-request latency. Fingerprints not found there fall through to the local catalogue of hashes I captured directly from specific SDK versions, and from there to heuristic rules.

The proxy and both backends run as containers on a virtual private server. The proxy is the only process exposed to the public internet — it handles TLS termination, computes the fingerprint, and forwards plain HTTP to whichever backend the classification calls for. Neither backend is reachable directly. The diagram below shows how incoming traffic moves through the system.

The Catalogue

Routing browser versus non-browser traffic is the easy case. The more interesting question is how much finer the classification can actually get. To find out, I ran the same HTTPS request from every runtime I could get my hands on — Python, Go, Rust, Node, and curl across both Linux and macOS — capturing the ClientHello from each. The table below shows what came back.

Most entries came back unambiguous. Go's crypto/tls implementation doesn't resemble anything else in the table — the combination of cipher suites and extensions it sends is distinct enough that a match is a reliable identification. The Python row covers every AI SDK currently in common use: the Anthropic client, the OpenAI client, LangChain all route through the same OpenSSL stack, so they land on the same row. The amber entries are where confidence drops. Rust on Linux and curl on Linux are similar enough that the classifier cannot distinguish them reliably. curl on macOS links against LibreSSL rather than OpenSSL, which makes it a completely different fingerprint from its Linux counterpart, so treating curl as a single identity would misclassify half that traffic.

The /stats page shows how the proxy has classified every connection since the site went live. The three categories — browser, agent, and tool — correspond to TLS library families. Chrome is recognizable immediately, because BoringSSL's fingerprint is distinct from every programmatic client in the catalogue. The agent/tool split is less tidy: agent covers clients whose fingerprints are consistent with a Python or Node runtime, and tool covers clients like curl. That split rests on a narrower signal than the labels imply.

The agent/tool split rests on a single number: how many TLS extensions the client sends. curl typically sends 7 or 8; Python's HTTP library typically sends 9 to 12. For every client in the catalogue, the match is exact and the extension-count range is consistent across every capture. That range is the entire basis for the classification.

What none of this data shows is the application. TLS fingerprinting identifies the library that made the connection, not what is running on top of it. The SDK, the tool name, the user, the intent are all absent at the transport layer. For clients in the catalogue, the library match is a reliable proxy for client class. For everything outside the catalogue, the classification is a reasonable inference, not an identification.

The fingerprint belongs to the library, not the application. Which SDK, which tool, which intent — none of that is visible at the transport layer. Classification above the library level is inference.

The gaps surface too. Fingerprints not matched by ja4db or the local catalogue fall back to heuristic classification — cipher count and extension count alone. The /stats page flags those connections; they are the ones worth capturing next.

Two layers, two different problems

The Cloudflare readiness checker and a TLS fingerprinting proxy are solving adjacent problems, not the same one. The readiness checker audits what you, the operator, have configured — robots.txt, markdown serving, MCP exposure. The fingerprinting proxy observes what arrived at connection time, before any of that configuration came into play. A site can be perfectly prepared to serve agents and receive nothing but scrapers. A proxy can classify every connection accurately and still serve content that agents cannot use. Each question is independently answerable. Neither answer is useful without the other.

Cloudflare's edge receives every ClientHello from every connection that passes through their network. JA4 fingerprinting is already part of their bot management stack, running at the edge and surfacing as a signal that Enterprise customers can build detection rules against. The agent readiness checker from this post's opening also runs there, as a configuration audit for origin operators. These are two separate systems that do not yet inform each other. The readiness checker runs at configuration time against static properties of the origin; the fingerprint runs at connection time against the client. These two signals sit inside the same product, indexing the same traffic, and they have not yet been connected.

The infrastructure to act on both signals simultaneously already exists inside a single product. An origin flagged as agent-ready could receive a connection fingerprinted as a Python HTTP client, and the routing decision — serve structured data, expose the MCP endpoint, skip the bot interstitial — could be made from both facts at once, before a single HTTP byte has passed. The readiness checker and the fingerprint already arrive at the same edge, on every connection. The infrastructure is already there. What it hasn't been used to do is route a connection using both facts at once.
